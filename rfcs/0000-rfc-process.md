## 摘要

本文规定 MindHub RFC 的角色、状态、元数据、章节、修订和 Git 记录要求。所有后续 RFC 都必须遵守本流程。

## 适用范围

凡是建立或改变 MindHub 的产品边界、核心语义、跨组件接口、治理规则或长期架构决策的提案，都应使用 RFC。局部实现细节、缺陷修复和不改变既有设计含义的维护工作不要求 RFC。

## 角色与决策权

RFC 作者负责起草提案、明确依赖、回应评审意见并维护修订历史。项目所有者 gittuyn 是唯一的最终决策者，负责决定 RFC 是否 Accepted 或 Rejected。投票、champion、申诉和强制定期复审不属于当前流程；改变这些规则必须通过新的 RFC。

## 状态模型

规范状态序列为 `Draft → Review → Accepted / Rejected → Superseded`。

- `Draft`：作者准备中的提案，可自由修改，尚未进入正式评审。
- `Review`：正在限期评审的提案；进入该状态时必须填写明确的评审截止日期。
- `Accepted`：最终决策者批准了设计。Accepted 只表示设计获批，不表示已经实现、发布或部署。
- `Rejected`：最终决策者拒绝了提案。文件必须保留在仓库中，并在“决策”章节记录拒绝理由。
- `Superseded`：已有新的 RFC 取代该 RFC 的规范作用；旧文件必须指明取代它的 RFC。

允许的转换及条件如下：

| 转换 | 条件 |
| --- | --- |
| Draft → Review | 作者认为提案可评审，并补充评审截止日期。 |
| Review → Accepted | 最终决策者批准设计，并记录决策日期与理由。 |
| Review → Rejected | 最终决策者拒绝设计，并记录决策日期与理由。 |
| Accepted → Superseded | 新 RFC 被 Accepted，且明确声明取代本 RFC。 |
| Rejected → Superseded | 新 RFC 对被拒方案作出替代性处置并被 Accepted，且明确声明取代本 RFC 的历史决策。 |

不得跳过 Review 直接从 Draft 进入 Accepted 或 Rejected。仅 RFC-0000 作为建立 RFC 流程本身的启动文档享有一次性例外，由最终决策者直接批准；后续 RFC 不得绕过 Review。Superseded 是终态；后续变化必须由新的 RFC 承担。

## RFC 生命周期

作者从 Draft 开始，完善必需元数据和章节后，将 RFC 改为 Review 并设定评审截止日期。评审期间，作者回应意见并记录实质修订。评审截止时，或最终决策者明确提前结束评审后，由最终决策者作出 Accepted 或 Rejected 决定并记录理由。若以后出现取代既有决定的新设计，应创建新 RFC；新 RFC 被 Accepted 后，旧 RFC 转为 Superseded 并互相链接。

## 编号与文件命名

RFC 使用不可复用的四位十进制编号，文件名格式为 `NNNN-小写连字符标题.md`。RFC-0000 保留给本流程。模板文件不占编号。编号一经分配，即使 RFC 被拒绝也不得重新使用。

## 必需元数据

每份 RFC 必须包含 RFC 编号、标题、作者、状态、创建日期、评审截止日期、最终决策者和依赖。Review 状态的评审截止日期必须是明确日期；Draft 可写“不适用（Draft）”。Accepted 或 Rejected RFC 还必须记录决策日期。依赖必须显式列出相关 RFC 编号；没有依赖时写“无”，不得留空或使用含糊占位符。

## 必需章节

除流程 RFC 可按治理需要调整外，每份 RFC 必须包含：摘要、动机、目标、非目标、术语、详细设计、考虑过的替代方案、安全、隐私与治理、兼容性、风险与开放问题、决策、修订历史。章节可以说明“不适用”及理由，但不得无解释地省略。

## 修订规则

Draft 可自由修订。Review 期间任何会改变范围、规范行为、接口、风险判断或决策依据的实质变更，都必须在“修订历史”中记录日期和摘要。拼写、排版、链接修复以及不改变含义的措辞澄清属于编辑性更正，可直接修改；改变要求、边界、语义或决策结果属于语义变更。

Accepted RFC 可接受编辑性更正，但不得通过原地改写引入重大语义变更。对 Accepted RFC 的重大变更必须创建新的 superseding RFC，并在新旧文件中明确取代关系。

## 决策记录

进入 Accepted 或 Rejected 时，“决策”章节必须记录最终状态、决策者、决策日期和可审查的理由。Rejected RFC 必须保留拒绝理由；Superseded RFC 必须记录取代它的 RFC 编号及该决定生效的日期。

## Git 要求

RFC 文件及其状态、决策和实质修订必须进入 Git 历史。提交应聚焦于可审查的 RFC 变化，并使索引、依赖、链接、状态和修订历史保持一致。不得删除 Rejected 或 Superseded RFC 来抹去决策历史。

## 替代方案

考虑过通过非正式文档或投票制管理设计决策。非正式文档无法稳定记录状态、依赖和决策理由；投票、champion 与申诉机制则超出项目当前规模。因此采用由作者提案、gittuyn 最终决策的轻量 RFC 流程。

## 安全、隐私与治理

所有 RFC 都必须分析其安全、隐私与治理影响，包括权限边界、敏感数据、可追溯性、自动化权力和人工控制。若某方面不适用，必须说明理由。治理权的改变本身属于 RFC 决策，不得仅以实现变更完成。

## 决策

本 RFC 于 2026-07-14 由 gittuyn 接受，立即成为 MindHub 后续 RFC 的规范流程。该接受表示流程设计获批，不表示任何其他 RFC 或产品功能已经实现。
