## 摘要

MindHub 是一个 AI Agent 优先参与、由人类最终掌控的开放共享知识中枢，用于沉淀、治理、演化和交换具有来源、范围及历史的长期知识。

本 RFC 确立 MindHub 的愿景、产品边界、核心原则及成功标准，并约束后续语义、生命周期、接口和实现 RFC。本文不规定知识实体结构、生命周期状态、删除语义、接口形态或传输协议。

## 动机

AI Agent 会从对话、工具调用、文档和协作活动中持续产生可复用信息，但现有能力通常分别关注执行期记忆、检索、存储或个人笔记，难以共同回答长期知识由谁产生、适用于何种范围、如何演变、能否撤销以及谁拥有最终控制权。

MindHub 需要先建立稳定的产品责任边界，使多 Agent 与人类能够共享知识而不丢失来源、适用范围和历史，并使自动化治理受到明确授权与人工控制。若没有这层边界，后续 RFC 容易把某种实体结构、治理机制或接入方式误当成产品本身。

## 定位

MindHub 面向 AI Agent 与人类的长期知识协作。它所称的 Knowledge Operating System（知识操作系统）是一层责任边界和治理层：统一承接长期知识的来源、范围、历史、权限与治理责任，并为不同实现和集成提供共同约束。

“知识操作系统”不表示 MindHub 取代通用操作系统、存储系统、AI 模型或传输机制，也不承诺某一种技术实现。开放意味着参与方可以检查和实现公共约束，可以替换合规组件，并可以连同来源、范围、历史和治理元数据导入或导出知识，而不是被单一厂商或实现锁定。导出后的策略执行受后文所述信任边界约束。

## 目标

- 建立以长期 Knowledge 而非短期 Memory 为中心的产品定位。
- 让 AI Agent 默认参与知识摄取和低风险治理，同时让自动化权限可配置、可收回且可追责。
- 保留每项知识的来源、适用范围及演变历史，使其能够被审计、纠正和回退。
- 支持多个 Agent 与人类在明确范围内共享知识，防止知识越权传播或使用。
- 规定实现可替换的产品边界，使存储、模型、检索和传输能力可以独立演进。
- 为 RFC-0002 至 RFC-0005 及后续实现 RFC 提供可检查的上位约束。

## 非目标

- 不在本文定义 Knowledge 实体、类别、身份或关系结构。
- 不在本文规定知识的生命周期、状态转换、保留期限或删除语义。
- 不在本文规定 OpenMemory 的接口形态、消息格式、兼容规则或传输语义。
- 不选择具体存储、检索、模型、部署方式或技术栈。
- 不要求 MindHub 取代 Agent Memory、RAG、向量存储、笔记系统或 MCP。
- 不把任何认知科学类比规定为实现事实或正确性依据。

## 术语

- **Candidate Knowledge（候选知识）：** 由对话、文档、工具结果、日志或外部系统产生，尚未完成知识治理的输入材料。候选知识不因被摄取而自动成为已治理的 Knowledge。
- **Knowledge（知识）：** MindHub 中可被长期引用和治理的知识主张。知识主张必须显式表达其来源、适用范围、历史及认知状态；它可以被支持、质疑、推导或否定，不代表 MindHub 将其宣告为唯一真相。其具体实体、身份、证据和关系结构由 RFC-0002 定义。
- **Memory（记忆）：** Agent 的工作记忆、执行上下文或其他可向长期知识提供输入的互补能力，不等同于 MindHub 管理的全部 Knowledge。
- **Knowledge Operating System（知识操作系统）：** MindHub 对长期知识的来源、范围、历史、权限与治理承担统一责任的边界和治理层，不指代通用操作系统或某一种技术实现。
- **AI Agent 优先参与：** 在明确授权和可接受风险范围内，默认允许 Agent 摄取候选知识并执行低风险治理；不表示无限自治或自动接受全部建议。
- **人类最终掌控：** Agent 的自动化权力必须来源于明确的人类授权，并受治理域、身份、角色和知识范围约束。只有在对应治理域内具有明确权限的人类主体能够配置、限制和撤销自动化权力，检查、否决或纠正结果，停止后续自动化，并决定删除与保留；Agent 永远不拥有最终决定权。不同人类主体之间的权利优先级和具体执行状态由 RFC-0003 定义。

## 详细设计

本 RFC 的规范性设计由后续章节共同构成：“定位”规定产品责任边界，“核心原则”规定必须持续满足的上位约束，“AI Agent 优先，人类最终掌控”规定自动化与人工权力关系，“系统边界”区分核心责任、互补组件和集成机制，“对后续 RFC 的约束”规定 RFC-0002 至 RFC-0005 及实现 RFC 必须遵循的边界。具体实体结构、生命周期与删除语义、接口与传输语义、风险分级与自动治理仍分别留给这些后续 RFC。

## 核心原则

1. **Knowledge First。** MindHub 管理可长期复用和治理的知识；Memory 可以是来源或互补能力，但不等同于产品边界。
2. **来源、范围和历史必须显式表达。** 系统必须记录已知信息，并能够表示 `unknown`、`partial`、`inferred` 或存在争议的状态；不得把缺失来源伪装成确定来源，也不得把推断结果伪装成直接观察。
3. **默认自动化，风险分级。** AI Agent 应默认承担摄取与明确授权范围内的低风险治理；风险越高，所需授权、审查和人工介入越强。
4. **人类最终掌控。** 删除权与数据主体权利完全属于人类；Agent 的权力只能来自人类授权，且不得自行扩大。不同人类主体之间对删除、保留和数据主体权利的优先级及执行状态由 RFC-0003 定义。
5. **多 Agent 共享不等于无边界共享。** 任何读取、写入、治理或传播都必须服从明确的身份、授权与范围约束。
6. **持续演化不抹去责任。** 知识可以被补充、合并、纠正或重新理解，但演化不得消除来源、行为者和历史。
7. **开放且实现可替换。** 产品语义不得依赖单一模型、存储、检索方式、传输机制或供应商。
8. **工程依据优先。** DIKW、脑区及其他认知科学比喻只可作为非规范性灵感，不构成术语定义、架构证明或实现要求。
9. **允许不确定与冲突。** MindHub 不充当唯一真相裁判。相互支持、矛盾或适用于不同时间、主体和情境的知识主张可以同时存在；系统必须保留其证据、范围、认知状态和演变历史。

## AI Agent 优先，人类最终掌控

“AI Agent 优先”表示在获得授权且风险可接受时，系统默认允许 Agent 摄取候选知识，并执行分类、关联、去重、摘要、冲突发现等低风险治理活动。它不表示 Agent 拥有无限自治，也不表示所有建议都应自动生效。

自动化权力必须按参与者、行为、知识范围和风险等级配置。系统必须支持升级至人工判断，也必须允许人类随时限制或撤销授权。每项自动行为及其影响应可审计；错误结果应可纠正，在适用时可回退；高风险或超出授权的行为不得以默认自动化绕过人工控制。

“人类最终掌控”表示自动化权力必须来源于明确的人类授权，并受治理域、身份、角色和知识范围约束。MindHub 不假定存在天然控制全部知识的单一人类；资源所有者、系统管理者、知识贡献者和数据主体可能拥有不同权利。Agent 永远不拥有删除权或数据主体权利，也不得成为最终治理权来源。

在相应治理域内具有明确权限的人类主体能够检查治理结果、否决或修正 Agent 决定、停止后续自动化，并决定删除与保留。删除可能涉及知识内容、个人数据、索引、派生结果或审计记录；不同对象可能受到隐私权、共享责任、安全审计和法定留存约束。RFC-0003 必须定义人类主体之间的权利优先级、删除与保留语义及执行状态，不得用无限期逻辑保留架空合法删除要求，也不得以彻底擦除为由破坏必要且合法的最小审计记录。

## 系统边界

MindHub 负责长期知识的共同责任边界：接纳来自不同参与者和集成的知识输入，保留来源与行为者线索，约束适用范围，承载可治理的演变历史，并向获授权参与者提供交换能力。

这些输入首先是 Candidate Knowledge；摄取只表示进入治理流程，不表示 MindHub 已将其接受为 Knowledge 或判定为事实。知识主张在完成相应治理后仍可以保持不确定、相互冲突或仅在特定范围内成立。

以下能力与 MindHub 互补，但都不等同于 MindHub 本身：

- Agent Memory 负责 Agent 的工作记忆或执行上下文，可以向 MindHub 提供输入或使用长期知识。
- RAG 负责以检索结果增强生成，可以消费 MindHub 管理的知识，但检索流程不是完整的知识治理。
- 向量存储负责相似性表示或检索，是可替换的实现组件。
- 笔记系统面向人的记录和组织，可作为导入、导出或同步集成。
- MCP 是可选的接入机制之一，不定义 MindHub 的产品语义。

同样，模型、规则引擎、索引、存储和传输机制可以帮助实现 MindHub，但任何单一组件都不能独自满足本 RFC 所规定的责任。

## 对后续 RFC 的约束

- RFC-0002 负责定义 Knowledge 实体及其身份。其设计必须使来源、范围、行为者和历史可表达，不得把短期 Memory 或某种存储记录直接等同于全部 Knowledge。
- RFC-0003 负责定义知识生命周期、删除与保留语义，以及不同人类主体之间的数据权利优先级和执行状态。其设计必须支持审计、纠正、适用时的回退和风险升级，并确保 Agent 不拥有删除权或数据主体权利。
- RFC-0004 负责定义 OpenMemory 接口与传输语义。其设计必须执行身份、授权和范围约束，不得将某一种传输机制规定为 MindHub 本身。
- 所有实现 RFC 必须说明其职责归属，并证明关键实现可替换；不得用实现选择反向改变本 RFC 的产品边界。
- RFC-0005 负责定义风险分级、自动治理、人工升级和人类主体之间的权限冲突处理。其设计不得赋予 Agent 最终治理权，也不得改变 RFC-0003 对删除权和数据主体权利优先级及执行状态的定义责任。

本文引用的 DIKW、海马体—新皮层或“数字新皮层”等类比均为非规范性灵感。后续 RFC 必须以可验证的工程理由定义行为，不得以类比代替领域定义、生命周期规则或协议语义。

## 成功标准

后续设计与实现只有同时满足以下可评审条件，才符合本 Manifesto：

1. **责任可分类：** 对任何提议能力，评审者能够明确判断它属于 MindHub 核心治理责任、互补组件、集成机制或范围外事项；不存在只能依赖具体实现才能解释的模糊归属。
2. **来源与行为者可追溯：** 对任何影响当前知识的摄取或治理行为，获授权的审计者能够确定输入来源、执行行为者及其导致的变化。
3. **多 Agent 范围可执行：** 两个具有不同身份或授权范围的 Agent 处理同一知识请求时，系统能够执行相应的读、写、治理和传播边界，并记录被拒绝的越界行为。
4. **风险可升级：** 超出授权或达到配置风险阈值的治理行为不会静默生效，而会停止、拒绝或升级给具有相应权限的人类。
5. **实现可替换：** 替换模型、存储、检索或传输实现时，只要新实现满足规范接口与治理约束，就不需要改变 Knowledge 的产品含义或放弃可追溯历史。
6. **人类可最终干预：** 具有相应权限的人类能够撤销 Agent 权限、停止自动化、检查和纠正结果、发起适用的回退，并对删除与保留作出最终的人类决定；系统能够留下这些干预的责任记录。
7. **低风险治理可自动完成：** 在明确授权的低风险场景中，系统能够在无需逐项人工批准的情况下完成常规治理，并记录自动执行、拒绝和升级结果；具有相应权限的人类可以调整或关闭该自动化。

## 考虑过的替代方案

**维持分散工具现状。** 可以继续由 Agent Memory、检索、存储和笔记工具各自承担局部能力，但无法形成一致的来源、范围、历史和治理责任，因此不采用。

**将 MindHub 定义为某一种 Memory 或检索产品。** 该定位易于实现，却会把长期知识治理缩减为保存或召回，并使多 Agent 权限、演变和人工控制成为附属能力，因此不采用。

**完全人工治理。** 它能够提供强控制，但无法承受 Agent 持续产生的大量候选知识，也会把人类变成逐项审批者，因此只在风险或授权边界要求时引入人工判断。

**完全自治治理。** 它可以降低操作成本，但缺少足够的升级、撤销、纠正，以及由人类决定删除与保留的保障，不符合人类最终掌控原则，因此不采用。

**把特定架构或认知类比写成产品定义。** 这会提前锁定尚待后续 RFC 验证的设计，并降低实现可替换性，因此仅保留为非规范性灵感。

## 安全、隐私与治理

共享长期知识会放大敏感信息泄露、错误传播、权限升级和自动化误判的影响。所有后续设计必须以最小必要权限执行摄取、读取、治理和交换；身份、授权、范围和风险判断必须可审查，敏感知识不得因多 Agent 协作而默认扩大可见范围。

治理行为必须关联可追溯的来源与行为者，保留足以解释当前结果的历史。自动化权限必须可配置、可升级、可撤销；系统必须支持审计、纠正和适用的回退，并确保具有相应权限的人类能够停止自动化及决定删除与保留。更具体的实体、生命周期和接口安全规则分别由 RFC-0002、RFC-0003 和 RFC-0004 规定。

来源、行为者、访问拒绝记录和修订历史本身也可能包含敏感信息。它们必须受到身份、范围、最小披露、保留期限和审计权限约束；“可审计”不表示所有参与者都能查看完整审计数据。

## 兼容性

本 RFC 依赖并遵循 RFC-0000 的流程与元数据要求，不改变既有代码、数据或接口。它为后续 RFC 建立上位产品约束，不表示那些设计已获接受或实现。

现有 Agent Memory、RAG、向量存储、笔记系统和 MCP 集成可以继续独立运行。未来接入 MindHub 时，应通过后续 RFC 规定的边界互操作，而无需被 MindHub 的某一实现替代。若未来提案需要改变本 Manifesto 的规范原则，应按 RFC-0000 创建新的取代性 RFC，而不能在实现 RFC 中隐式改写。

导入和导出必须携带能够表达来源、范围、历史和治理要求的元数据。MindHub 只能在自身及受信任协议的控制边界内强制策略；数据离开该边界后，接收方是否继续执行权限、撤销和删除要求，必须由协议能力、信任关系或外部契约保证，不得宣称 MindHub 能对任意外部副本保持绝对控制。

## 风险与开放问题

- **原则过于抽象：** 可能导致不同实现声称兼容却行为不一。缓解方式是要求 RFC-0002 至 RFC-0005 将本 RFC 的成功标准转化为可验证规则。
- **自动化与人工控制失衡：** 过度升级会使治理不可用，升级不足会扩大错误影响。风险阈值和授权机制留给后续治理设计，但不得削弱本文的升级、撤销和最终控制要求。
- **开放与互操作成本：** 实现可替换可能增加规范工作。后续接口 RFC 应优先定义稳定语义，而非绑定具体技术。

开放问题限定为：风险等级如何表达、哪些行为允许自动生效、人工升级所需的证据集合，以及多人权利冲突如何裁决。知识主张与证据由 RFC-0002 具体化；删除权、数据主体权利的优先级及执行状态由 RFC-0003 具体化；跨系统信任边界由 RFC-0004 具体化；风险分级、自动治理、人工升级和权限冲突处理由 RFC-0005 具体化。这些 RFC 必须依据威胁分析、原型验证和典型多 Agent 场景作出决定，不得削弱本文对人类权利、可配置授权、风险升级和人工最终控制的要求。
